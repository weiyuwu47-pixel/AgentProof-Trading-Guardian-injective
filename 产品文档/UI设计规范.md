# 芙副官 UI 设计规范

本文档用于统一 `芙副官 / AgentProof Trading Guardian` 后续前端页面、演示物料和产品截图的视觉方向。目标是让产品看起来像可信的 AI 金融基础设施，而不是交易娱乐平台或乙游页面。

## 1. 设计定位

### 1.1 产品气质

芙副官是 AI 金融 Agent 的可验证决策层。UI 应传达：

- 可验证：用户能快速理解决策经过验证。
- 可追溯：决策链路、行为日志、proof 和 anchor 可被追踪。
- 可信任：产品克制、清晰、专业，不制造收益焦虑。

### 1.2 视觉关键词

- 浅蓝
- 透明玻璃
- 金融科技
- 验证感
- 守护者
- 冷静可信
- 轻量科幻

### 1.3 明确避免

- 不做乙游界面。
- 不做恋爱聊天界面。
- 不使用暧昧、亲密、陪伴感过强的表达。
- 不做赌场、暴富、收益承诺式交易 UI。
- 不做深紫、暗黑、赛博朋克压迫感。
- 不在首页堆 dashboard、hash、交易图表和过多技术细节。

## 2. 品牌与文案

### 2.1 品牌显示

品牌名应同时保留：

```text
芙副官
AgentProof Trading Guardian
```

中文品牌 `芙副官` 优先级最高，英文名作为副标题或技术品牌名出现。

### 2.2 首页核心文案

首页只负责建立第一印象、讲清楚产品价值、让用户选择入口。首页保留：

```text
芙副官
AgentProof Trading Guardian
AI 金融 Agent 的可验证决策层
验证 AI 是否读取真实数据、遵守策略、通过风控，并将 proof 锚定到 Injective。
```

两个入口按钮：

```text
普通用户，体验虚拟运算流程
我是开发者，部署到本地自行接入真实 API
```

三个核心价值点：

```text
可验证
可追溯
可信任
```

### 2.3 页面语言

面向用户的可见文案以中文为主。允许保留必要英文技术词：

- Agent
- proof
- hash
- Injective
- testnet
- API
- CLI
- verifier

技术命令、环境变量、仓库名保持英文原文。

## 3. 色彩系统

### 3.1 主色

```text
Primary Blue: #2f6fed
Deep Navy: #12316c
Sky Blue: #8fc7ff
Ice Blue: #eaf6ff
```

### 3.2 背景色

```text
Page Background: #f3f9ff
Panel Background: rgba(255, 255, 255, 0.72)
Panel Strong: rgba(255, 255, 255, 0.88)
```

### 3.3 状态色

```text
Success: #16a56b
Warning: #f08a3c
Rejected: #e5533d
Muted Text: #6f7f99
```

### 3.4 使用原则

- 浅蓝和白色是主视觉。
- 深蓝用于标题、重要数字和导航状态。
- 绿色仅用于 `PASSED`、验证通过、锚定成功。
- 红橙色仅用于幻觉检测、拒绝、风险提醒。
- 不使用大面积紫色、黑色或高饱和渐变。

## 4. 版式系统

### 4.1 圆角与卡片

- 标准圆角：`8px`
- 大型页面容器：`12px`
- 不使用过度圆润的可爱风格。
- 卡片应轻薄、透明、带细边框。
- 避免卡片套卡片；需要分组时使用分割线、网格或浅色块。

### 4.2 首页布局

首页是 Landing Page，不是 dashboard。

推荐结构：

```text
左侧：
  品牌名
  一句话价值主张
  一句解释
  两个入口按钮
  三个价值点

右侧：
  傅融参考方向的“副官 / 守护者”人物形象
  浅蓝光柱、验证符号、抽象金融线条
```

首页不要展示：

- Proof hash
- Tx hash
- 大量 dashboard 卡片
- K 线图
- 长流程解释
- 技术路线图

### 4.3 普通用户体验页布局

优先级：

```text
1. 输入股票与策略
2. 验证进度
3. 验证结果
4. 查看可验证提醒
5. 更详细的 proof / behavior log / anchor
```

首屏推荐：

```text
左：输入表单
右：验证步骤
下：HOLD / PASSED / 可验证提醒
```

### 4.4 幻觉演示页布局

推荐结构：

```text
真实记录数据 → Agent 幻觉输出 → Verifier 检查
                            ↓
                      REJECTED 结果
```

红橙色只用于 `REJECTED` 和关键风险提示，其余仍保持浅蓝体系。

### 4.5 开发者页布局

开发者页应更工具化：

```text
左：项目仓库 + .env 安全提醒
右：运行命令
下：技术链路
```

开发者页可以显示代码块，但保持清爽，不要变成纯黑终端页。

## 5. 人物形象使用规范

### 5.1 角色定位

人物形象是“AI 金融副官 / 验证守护者”，不是恋爱对象。使用时应体现：

- 冷静
- 可靠
- 专业
- 守护
- 审计与验证

### 5.2 形象方向

可参考傅融的方向：

- 深色长发、高马尾轮廓
- 黑灰服装
- 蓝色点缀
- 清冷、克制、可信赖

但必须避免直接复制原图、水印、具体服装细节和乙游构图。

### 5.3 使用方式

- 首页可以放较大人物，作为第一印象。
- 子页面只作为右侧淡化背景、侧边辅助形象或小头像。
- 不使用人物对话气泡。
- 不使用亲密视角、半身大脸特写、恋爱约会氛围。
- 人物不能压过产品信息。

## 6. 组件规范

### 6.1 按钮

主按钮：

- 深浅蓝渐变
- 白色文字
- 左侧图标
- 右侧可用箭头

次按钮：

- 白色或玻璃背景
- 蓝色边框
- 深蓝文字

按钮文案要直接表达入口或动作，不写营销口号。

### 6.2 表单

表单用于普通用户体验页：

- 输入框背景为白色半透明。
- 标签使用短中文。
- 数字输入不要过度装饰。
- 策略文本框要足够大，支持自然语言。

### 6.3 步骤流

步骤状态：

- 未开始：浅灰蓝
- 运行中：主蓝
- 已完成：绿色或蓝色对勾
- 拒绝：红橙色警示

步骤文案应短：

```text
读取用户策略
模拟读取市场数据
Agent 生成交易决策
执行风控检查
生成 proof hash
Verifier 验证
```

### 6.4 结果区

普通用户优先看到：

```text
600941 / 中国移动
HOLD
PASSED
查看用户收到的可验证提醒
```

不要让普通用户一开始看到长 hash。

### 6.5 技术链路

开发者页可展示：

```text
Agent Identity
Stable JSON Hashing
Behavior Log
Injective Anchor
Local Verifier
```

技术链路使用横向 chip / node，不要做复杂架构图。

## 7. 图标规范

推荐使用线性图标：

- shield：验证、可信、安全
- check：通过
- alert：拒绝、风险
- terminal：开发者命令
- copy：复制命令或 hash
- database / file：行为日志、proof package

图标风格应统一为线性、浅蓝、低重量。避免夸张 3D 图标和拟物图标。

## 8. 页面清单

### 8.1 首页

目标：

- 建立第一印象
- 说明产品价值
- 让用户选择入口

核心元素：

- 品牌名
- 价值主张
- 傅融方向的副官形象
- 两个入口按钮
- 三个价值点

### 8.2 普通用户体验页

目标：

- 输入策略
- 体验虚拟验证流程
- 查看验证结果和可验证提醒

核心元素：

- 输入表单
- 验证步骤
- HOLD / PASSED 结果
- 可验证提醒弹窗

### 8.3 幻觉演示页

目标：

- 展示 Agent 编造行情时如何被拒绝

核心元素：

- 真实记录数据
- Agent 幻觉输出
- Verifier 检查
- REJECTED 结果

### 8.4 开发者页

目标：

- 引导开发者本地部署
- 接入真实 API / Injective testnet

核心元素：

- GitHub 仓库
- `.env` 安全提醒
- CLI 命令
- 技术链路

## 9. 可访问性与响应式

- 所有按钮必须有明确可读文案。
- 文本与背景对比度要足够。
- 移动端优先纵向堆叠，人物形象降级为背景或隐藏。
- 文案不能溢出按钮和卡片。
- 长 hash、命令行要使用省略或换行处理。
